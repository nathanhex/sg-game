# 倒水游戏（Tube Pour）关卡数据说明

本文档说明倒水游戏的关卡设置文件位置、数据结构含义及校验规则，供修改关卡或编写生成工具时参考。

---

## 1. 关卡设置文件位置

| 用途 | 路径 | 说明 |
|------|------|------|
| **关卡数据主文件** | `games/tube-pour/js/game.js` | 游戏逻辑与关卡数据均在此文件中。关卡以 `LEVELS` 数组形式内联定义（约第 20 行起）。 |
| **关卡生成工具** | `games/tube-pour/tools/generate-100-levels.js` | 生成 100 关数据，输出可替换 `game.js` 中的 `LEVELS` 数组。 |
| **关卡生成工具（改进版）** | `games/tube-pour/tools/generate-100-levels-improved.js` | 同用途，算法或参数可能不同。 |
| **关卡诊断工具** | `games/tube-pour/tools/diagnose-levels.js` | 从 `game.js` 中提取并分析 `LEVELS`，检查可解性、单色管比例等。 |

**修改关卡时**：直接编辑 `games/tube-pour/js/game.js` 中的 `LEVELS` 数组；或运行生成工具，将其输出粘贴替换该数组。

---

## 2. 单关数据结构（Level 对象）

每一关是一个普通对象，包含以下字段：

| 字段 | 类型 | 必填 | 含义 |
|------|------|------|------|
| `id` | number | 否（建议有） | 关卡编号，用于选关、诊断等。随机/自定义关可用 `-1`。 |
| `capacity` | number | **是** | 每根试管的容量（格子数）。当前全部关卡为 `4`。 |
| `tubes` | number[][] | **是** | 试管初始状态，见下文。 |
| `colors` | string[] | 否 | 该关使用的颜色数组（十六进制，如 `"#FF6B6B"`）。缺省时使用 `DEFAULT_COLORS`（前 4 色）。 |

### 2.1 `tubes` — 试管初始状态

- **类型**：数组的数组，即「多根试管，每根试管是一个数组」。
- **长度**：至少 2 根试管（否则 `validateLevel` 判为无效）。
- **每根试管**：一个**数字数组**，表示从**底部到顶部**每格液体的**颜色编号**。
  - 颜色编号为 **1-based**：`1` = 本关 `colors[0]`，`2` = `colors[1]`，以此类推。
  - 空试管用空数组 `[]` 表示。
  - 每根试管内元素个数不能超过 `capacity`。

**示例**（capacity=4，2 色）：

```js
tubes: [
  [1, 2, 1],   // 试管 0：底→顶 为 色1、色2、色1
  [2, 1, 2],   // 试管 1：底→顶 为 色2、色1、色2
  []           // 试管 2：空
]
```

---

## 3. 颜色与编号对应关系

- **`colors`**：字符串数组，如 `["#FF6B6B", "#4ECDC4", "#95E86C"]`。
- **试管中的数字**：1 对应 `colors[0]`，2 对应 `colors[1]`，3 对应 `colors[2]`……
- 渲染时使用：`colors[colorId - 1]`（见 `game.js` 中 `renderTubes`）。因此**严禁使用 0**，必须从 1 开始连续编号到本关颜色数。

游戏内全局调色板 `PALETTE` 共 12 色，前 4 色与 `DEFAULT_COLORS` 一致；生成工具用 `PALETTE.slice(0, numColors)` 得到本关 `colors`。

---

## 4. 校验规则（validateLevel）

游戏加载关卡时会调用 `validateLevel(level)`，以下条件**全部满足**才视为有效：

1. **对象与 tubes**：`level` 存在、`level.tubes` 存在且为数组，且 **tubes.length ≥ 2**。
2. **容量**：每根试管的 `tubes[i].length` **≤** `level.capacity`，不得超载。
3. **颜色数量**：每种颜色编号（即试管中出现的数字）在**全部试管**中的出现次数 **≤ capacity**。  
   （即：每种颜色最多只能「装满」一整管，不能多出一格。）
4. **至少一根非满管**：至少有一根试管未满（`tubes[i].length < capacity`）。  
   若全部试管都满，则必须已经是过关状态（见下条）。
5. **非即过关**：若存在非满管，不要求当时已过关；若**全部试管都满**，则当前状态必须已满足过关条件，否则视为无效（避免「开局即过关」或矛盾状态）。

过关条件（`isLevelComplete`）：  
每根试管要么为空，要么整管同色；且**同一种颜色只能出现在一根试管里**（即每种颜色恰好占满若干整管）。

---

## 5. 游戏规则与 capacity

- **倒水规则**：只有试管顶部一段**同色**液体可以一起倒出；只能倒入**空试管**或**顶部颜色相同**的试管；一次倒入的量不超过目标管剩余空间，也不超过源管顶部同色连续段长度。
- **capacity**：当前版本所有关卡均为 `capacity: 4`。若将来改为 5 或 6，需保证 `tubes` 中每管长度 ≤ 新 capacity，且颜色数量规则不变（每种颜色出现次数 ≤ capacity）。

---

## 6. 示例：完整一关

```js
{
  id: 1,
  capacity: 4,
  tubes: [
    [1, 2, 1],    // 管0：3 格，顶为色1
    [2, 1, 2],    // 管1：3 格，顶为色2
    []            // 管2：空
  ],
  colors: ["#FF6B6B", "#4ECDC4"]
}
```

- 2 种颜色，编号 1、2，对应两种十六进制色。
- 3 根试管，容量均为 4；有两根非满、一根空，满足「至少一根非满」。
- 每种颜色出现次数 ≤ 4，且未过关，校验通过。

## 7. 生成工具要点（可解性）

- **可解性**：当前权威方案改为「**候选局面生成 + 正向求解器验收**」。
  - **候选局面**：生成器可以直接构造混合开局，也可以从某个中间状态演化得到候选局面；但候选局面本身必须满足本文档第 4 节的结构与校验规则。
  - **求解器验收**：每个候选局面都必须经过正向规则求解验证。只有当求解器能从该开局找到一条合法解路径时，才允许接受该关。
  - **显式禁止的误用**：如果采用“过关状态 + 反向构造”，则必须实现并验证**真正的逆操作**。**禁止**直接复用正向倒水合法性来充当“反向倒水”，因为从过关状态出发，这种伪逆操作通常只能把整段单色液体倒入空管，无法有效生成混合局面。
- 工具内 `validateLevel` 与 `game.js` 中的校验逻辑一致（试管数、容量、颜色数量、非即过关等）；生成后可用 `diagnose-levels.js` 做进一步分析，并建议在工具链中增加求解验证。

### 7.1 推荐生成流程

推荐按以下顺序生成关卡：

1. 根据关卡档位确定 `numColors`、`numTubes`、`capacity` 与目标难度约束。
2. 直接构造一个**候选开局**，确保颜色计数正确、至少一根试管未满，且不是开局即通关。
3. 对候选开局施加混合度约束，例如限制高难档的最大连续同色长度、减少单色管数量。
4. 用与游戏一致的**正向倒水规则**运行求解器；只有求解成功的候选局面才可保留。
5. 如有多个候选局面可选，再按混合度、已归位管数量、可选最短解长度等指标进行筛选。

### 7.2 混合度与难度（质量门控）

为避免高难度关卡因「大量试管连续 3 格同色」而过于简单，生成与验收时应考虑**混合度**：

- **单管最大连续同色长度**：一根试管内同一颜色连续出现的最大长度。困难/非常困难档（如 id ≥ 56）要求该值 **≤ 2**（即不允许连续 3 格同色）；简单/中等可放宽。
- **整管同色数**：满足「长度 ≥ 2 且整管同色」的试管数量，候选生成时应尽量少；困难/非常困难最终宜为 0。
- **求解验证优先级**：混合度约束用于提高难度质量，但不能代替可解性验证。最终是否接受关卡，以 `validateLevel` 通过、`isLevelComplete` 为 false、且求解器可解为准。

完整的关卡设计与生成规则（档位、候选生成策略、求解验收、质量门控、验收标准）见 **`docs/plans/2026-03-04-tube-pour-level-design-and-generation.md`**。
